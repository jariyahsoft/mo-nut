# 1. บทสรุปผู้บริหาร

หมอนัด หรือ **Mo-nut** เป็นแพลตฟอร์มช่วยดูแลผู้ป่วยโรคเรื้อรัง เช่น เบาหวาน ความดันโลหิตสูง โรคหัวใจ โรคไต ไขมันในเลือดสูง และผู้สูงอายุที่ต้องรับประทานยาหลายชนิดหรือไปพบแพทย์อย่างต่อเนื่อง

ผลิตภัณฑ์แก้ปัญหาสำคัญที่เกิดขึ้นระหว่างการดูแลผู้ป่วย ได้แก่

* ผู้ป่วยลืมวันนัด
* ทำใบนัดหาย
* ลืมรับประทานยา
* จำคำแนะนำของแพทย์หรือเภสัชกรไม่ได้
* ผู้ดูแลไม่ทราบตารางนัดหรือสถานะการรับประทานยา
* ข้อมูลความดัน น้ำหนัก น้ำตาล และผลตรวจถูกเก็บแยกกัน
* ผู้ป่วยลืมคำถามที่ต้องการถามแพทย์
* ผู้ป่วยไม่สามารถติดตามคำแนะนำด้านอาหารหรือการออกกำลังกายได้อย่างต่อเนื่อง
* การเดินทางไปโรงพยาบาลล่าช้าเนื่องจากสภาพจราจร
* ผู้ดูแลไม่สามารถช่วยเหลือได้ทันเมื่อเกิดเหตุฉุกเฉิน

หมอนัดจะรวบรวมการนัดหมาย การรับประทานยา ข้อมูลสุขภาพ เอกสารทางการแพทย์ คำแนะนำจากแพทย์ เช็กลิสต์การดูแลตนเอง การเดินทาง และการติดต่อผู้ดูแลไว้ในระบบเดียว

จุดเด่นของผลิตภัณฑ์คือการเชื่อมโยงผู้ใช้งาน 3 ฝ่ายเข้าด้วยกัน ได้แก่

1. ผู้ป่วย
2. ผู้ดูแลหรือสมาชิกครอบครัว
3. แพทย์และบุคลากรทางการแพทย์

โดยผู้ป่วยยังคงเป็นผู้ควบคุมสิทธิ์การเข้าถึงและการแชร์ข้อมูลสุขภาพของตนเอง


## 1.1 กลยุทธ์การพัฒนาผลิตภัณฑ์แบบ 2 เฟส

การพัฒนาหมอนัดแบ่งออกเป็น 2 เฟสหลัก เพื่อให้สามารถนำผลิตภัณฑ์ออกทดสอบกับผู้ใช้จริงได้เร็ว ลดความเสี่ยงด้านต้นทุน และต่อยอดไปสู่ประสบการณ์แอปมือถือที่เสถียรขึ้นโดยไม่ต้องเปลี่ยนระบบข้อมูลใหม่ทั้งหมด

| หัวข้อ | Phase 1 — Mobile-first PWA | Phase 2 — Cross-platform Mobile App |
| --- | --- | --- |
| เป้าหมาย | พิสูจน์ Product–Market Fit และความต้องการของผู้ป่วย/ผู้ดูแล | เพิ่มความน่าเชื่อถือ การใช้งานประจำวัน และความสามารถระดับอุปกรณ์ |
| ช่องทางใช้งาน | Browser, ติดตั้ง PWA บนหน้าจอโฮม, Tablet และ Desktop | แอป Android และ iOS ผ่าน Google Play/App Store พร้อมคง Web Portal |
| จุดเน้น | ฟังก์ชันหลัก นัดหมาย ยา สุขภาพ ผู้ดูแล OCR เสียง เช็กลิสต์ และรายงาน | Native Notification, Background Task, Biometric, Secure Storage, Widget, Health Platform และอุปกรณ์ Bluetooth |
| แนวทางเทคนิค | Mobile-first, Responsive, Progressive Enhancement และ Offline ขั้นพื้นฐาน | Cross-platform Codebase เชื่อม Backend/API และข้อมูลชุดเดียวกับ Phase 1 |
| ผลลัพธ์ทางธุรกิจ | ข้อมูลการใช้งานจริงและข้อพิสูจน์ว่าปัญหาหลักได้รับการแก้ไข | ผลิตภัณฑ์พร้อมขยายผู้ใช้ เพิ่ม Retention และรองรับพันธมิตร/องค์กร |

หลักสำคัญคือ **Phase 2 ต้องเป็นการต่อยอด ไม่ใช่การเริ่มสร้างผลิตภัณฑ์ใหม่ทั้งหมด** โดยบัญชีผู้ใช้ ข้อมูลสุขภาพ สิทธิ์การแชร์ API กติกาธุรกิจ และ Audit Log ต้องใช้งานต่อเนื่องจาก Phase 1 ได้

### เกณฑ์เข้าสู่ Phase 2

การเริ่มพัฒนา Phase 2 ควรเกิดขึ้นเมื่อผ่านเกณฑ์ต่อไปนี้

1. ผู้ใช้กลุ่มเป้าหมายยืนยันว่าฟังก์ชันนัดหมาย ยา และผู้ดูแลสร้างคุณค่าจริง
2. งานหลักของผู้ป่วยและผู้ดูแลผ่านการทดสอบ Usability กับผู้สูงอายุ
3. มีข้อมูล Retention, Notification Reliability และข้อจำกัดของ PWA เพียงพอสำหรับตัดสินใจ
4. Backend, API, Data Model, Consent และ Security พร้อมรองรับ Mobile App
5. มีแผนงบประมาณ การดูแล App Store/Google Play และการสนับสนุนเวอร์ชันแอประยะยาว

# 2. วิสัยทัศน์ผลิตภัณฑ์

> “ทำให้ผู้ป่วยทุกคนไม่พลาดนัด ไม่พลาดยา และมีคนช่วยดูแลในทุกช่วงของการรักษา”

หมอนัดมีเป้าหมายเป็นผู้ช่วยสุขภาพส่วนบุคคลที่อยู่กับผู้ป่วยตั้งแต่ก่อนพบแพทย์ ระหว่างพบแพทย์ หลังกลับจากโรงพยาบาล และตลอดช่วงเวลาที่ต้องดูแลตนเองที่บ้าน

---

# 3. พันธกิจ

1. ลดจำนวนผู้ป่วยที่พลาดนัดหมาย
2. เพิ่มความสม่ำเสมอในการรับประทานยา
3. ช่วยให้ผู้ป่วยจดจำและปฏิบัติตามคำแนะนำของแพทย์
4. ช่วยให้ครอบครัวและผู้ดูแลติดตามผู้ป่วยได้อย่างเหมาะสม
5. ทำให้ข้อมูลสุขภาพของผู้ป่วยเป็นระบบและนำไปใช้ประกอบการรักษาได้
6. ลดความซับซ้อนในการใช้งานแอปสุขภาพสำหรับผู้สูงอายุ
7. สร้างช่องทางเชื่อมต่อระหว่างผู้ป่วย ผู้ดูแล และสถานพยาบาล

# 4. ปัญหาที่ผลิตภัณฑ์ต้องการแก้ไข

## 4.1 ปัญหาด้านนัดหมาย

* ผู้ป่วยจำวันหรือเวลานัดผิด
* ใบนัดสูญหายหรือชำรุด
* ผู้ดูแลไม่ทราบว่าต้องพาผู้ป่วยไปโรงพยาบาลเมื่อใด
* ผู้ป่วยไม่ทราบว่าต้องเตรียมเอกสารหรืออดอาหารก่อนตรวจ
* ผู้ป่วยออกเดินทางช้าและไปไม่ทันนัด

## 4.2 ปัญหาด้านการรับประทานยา

* ผู้ป่วยลืมกินยา
* ไม่แน่ใจว่ากินยาไปแล้วหรือยัง
* จำลักษณะหรือชื่อยาไม่ได้
* กินยาผิดเวลา
* ยาใกล้หมดแต่ไม่มีการแจ้งเตือน
* ผู้ดูแลไม่ทราบว่าผู้ป่วยพลาดยา

## 4.3 ปัญหาด้านข้อมูลสุขภาพ

* ข้อมูลน้ำหนัก ความดัน น้ำตาล และผลตรวจไม่ได้ถูกจัดเก็บเป็นระบบ
* ผู้ป่วยไม่สามารถแสดงแนวโน้มข้อมูลให้แพทย์ดูได้
* ข้อมูลอยู่ในสมุด รูปภาพ โทรศัพท์ หรือเอกสารหลายแห่ง

## 4.4 ปัญหาด้านการสื่อสาร

* ผู้ป่วยจำคำแนะนำของแพทย์หรือเภสัชกรไม่ได้
* ผู้ป่วยลืมคำถามที่ตั้งใจจะถามแพทย์
* ผู้ดูแลไม่ได้อยู่ด้วยในวันที่ไปพบแพทย์
* ผู้ป่วยอธิบายประวัติหรืออาการย้อนหลังได้ไม่ครบถ้วน

## 4.5 ปัญหาด้านการปฏิบัติตามคำแนะนำ

* คำแนะนำด้านอาหารและการออกกำลังกายไม่มีระบบติดตาม
* ผู้ป่วยไม่ทราบว่าทำตามเป้าหมายได้กี่วัน
* แพทย์หรือผู้ดูแลไม่เห็นความคืบหน้าระหว่างนัด

## 4.6 ปัญหาด้านเหตุฉุกเฉิน

* ผู้สูงอายุไม่สามารถค้นหาเบอร์ผู้ดูแลได้ทัน
* ผู้ดูแลไม่ทราบตำแหน่งของผู้ป่วย
* ข้อมูลโรคประจำตัวและประวัติแพ้ยาไม่พร้อมใช้งานเมื่อเกิดเหตุ

# 5. กลุ่มเป้าหมาย

## 5.1 กลุ่มเป้าหมายหลัก

* ผู้ป่วยโรคเบาหวาน
* ผู้ป่วยความดันโลหิตสูง
* ผู้ป่วยโรคหัวใจ
* ผู้ป่วยโรคไต
* ผู้ป่วยไขมันในเลือดสูง

* ผู้ป่วยที่ต้องรับประทานยาหลายชนิด
* ผู้สูงอายุ
* ผู้ป่วยที่ต้องไปพบแพทย์เป็นประจำ
* ผู้ป่วยที่ต้องมีสมาชิกครอบครัวช่วยดูแล

## 5.2 กลุ่มเป้าหมายรอง

* โรงพยาบาล
* ร้านขายยา
* ศูนย์ดูแลผู้สูงอายุ

# 6. ผู้ใช้งานและบทบาท

## 6.1 ผู้ป่วย

ผู้ป่วยเป็นเจ้าของข้อมูลสุขภาพและเป็นผู้กำหนดสิทธิ์การแชร์ข้อมูล

ผู้ป่วยสามารถ

* จัดการนัดหมาย
* ถ่ายรูปใบนัด
* บันทึกข้อมูลสุขภาพ
* จัดการตารางยา
* ยืนยันการรับประทานยา
* บันทึกเสียงคำแนะนำ
* สร้างเช็กลิสต์
* บันทึกคำถามสำหรับแพทย์
* แชร์ข้อมูลให้ผู้ดูแลหรือแพทย์
* ใช้ปุ่ม SOS

## 6.2 ผู้ดูแล

ผู้ดูแลอาจเป็นสมาชิกครอบครัว ญาติ ผู้ดูแลประจำ หรือบุคลากรในชุมชน

ผู้ดูแลสามารถ

* ดูแลผู้ป่วยหนึ่งคนหรือหลายคน
* ดูวันนัด
* รับแจ้งเตือนก่อนถึงวันนัด
* รับแจ้งเตือนเมื่อผู้ป่วยพลาดยา
* ช่วยบันทึกข้อมูลแทนผู้ป่วย
* ดูแนวโน้มข้อมูลสุขภาพ
* ดูเช็กลิสต์ที่แพทย์แนะนำ
* รับการแจ้งเตือน SOS
* ช่วยจัดการการเดินทางไปสถานพยาบาล

## 6.3 แพทย์และบุคลากรทางการแพทย์

แพทย์หรือบุคลากรทางการแพทย์สามารถเข้าถึงข้อมูลได้เฉพาะเมื่อผู้ป่วยอนุญาต

ความสามารถที่วางแผนไว้ ได้แก่

* ดูข้อมูลสรุปสุขภาพ
* ดูประวัติความดัน น้ำหนัก และน้ำตาล
* ดูประวัติการรับประทานยา
* บันทึกคำแนะนำ
* สร้างเช็กลิสต์ให้ผู้ป่วย
* ระบุเป้าหมายด้านสุขภาพ
* เพิ่มวันนัดครั้งถัดไป
* เพิ่มหรือปรับปรุงรายการยา
* ดูคำถามที่ผู้ป่วยเตรียมมา
* ดูความคืบหน้าระหว่างนัด

## 6.4 ผู้ดูแลระบบ

ผู้ดูแลระบบสามารถ

* จัดการผู้ใช้งาน
* จัดการสถานพยาบาล
* ตรวจสอบสถานะระบบ
* จัดการเนื้อหาสุขภาพ
* ตรวจสอบ Audit Log
* จัดการคำร้องเรียน
* จัดการเหตุการณ์ด้านความปลอดภัย
* จัดการแพ็กเกจสมาชิก
* จัดการองค์กรที่ใช้ระบบ

# 7. คุณค่าที่ผลิตภัณฑ์มอบให้

## 7.1 สำหรับผู้ป่วย

* ไม่พลาดวันนัด
* ไม่พลาดยา
* จำคำแนะนำของแพทย์ได้
* เก็บเอกสารสุขภาพไว้ในที่เดียว
* ดูแนวโน้มสุขภาพของตนเองได้
* เตรียมคำถามก่อนพบแพทย์
* ขอความช่วยเหลือได้ง่ายเมื่อเกิดเหตุฉุกเฉิน

## 7.2 สำหรับผู้ดูแล

* ทราบสถานะของผู้ป่วยโดยไม่ต้องโทรถามตลอดเวลา
* ทราบเมื่อผู้ป่วยพลาดนัดหรือพลาดยา

# 12. แพลตฟอร์มที่รองรับ

## 12.1 ภาพรวมการพัฒนา 2 เฟส

| รายการ | Phase 1 — Mobile-first PWA | Phase 2 — Cross-platform Mobile App |
| --- | --- | --- |
| อุปกรณ์หลัก | Smartphone ผ่าน Browser/PWA | Smartphone ผ่านแอป Android/iOS |
| อุปกรณ์รอง | Tablet และ Desktop Browser | Tablet ตามความเหมาะสม และ Web Portal เดิม |
| การติดตั้ง | เพิ่ม PWA ลงหน้าจอโฮม | Google Play และ Apple App Store |
| การแจ้งเตือน | In-app และ Web Push เมื่อแพลตฟอร์มรองรับ | Native Push + Local Notification + Background Processing |
| Offline | อ่านข้อมูลล่าสุดและบันทึกคิวรอซิงก์แบบพื้นฐาน | Offline-first ที่แข็งแรงขึ้น พร้อมฐานข้อมูลในอุปกรณ์แบบเข้ารหัส |
| ความสามารถอุปกรณ์ | Camera, Microphone, Geolocation และ Web Share ตาม Browser | Biometric, Secure Storage, Widget, Shortcut, Health Platform, Bluetooth และ Deep Link ระดับระบบ |
| Backend และข้อมูล | ใช้ Backend/API กลาง | ใช้ Backend/API และข้อมูลชุดเดียวกับ Phase 1 |

## 12.2 Phase 1 — Mobile-first PWA

Phase 1 ต้องพัฒนาเป็น **Web Application แบบ Mobile-first Progressive Web App (PWA)** โดยออกแบบประสบการณ์สำหรับสมาร์ตโฟนเป็นหลัก และรองรับ Tablet กับ Desktop Browser แบบ Responsive

แพลตฟอร์มที่รองรับ ได้แก่

* Android Smartphone ผ่าน Browser และการติดตั้ง PWA
* iPhone ผ่าน Safari และการเพิ่ม PWA ไปยังหน้าจอโฮม
* Tablet ผ่าน Browser
* Desktop และ Laptop ผ่าน Browser
* Web Portal สำหรับผู้ดูแล แพทย์ในโครงการนำร่อง และผู้ดูแลระบบ

คุณสมบัติหลักของ PWA ได้แก่

* Responsive Design และ Mobile-first Layout
* Web App Manifest และ Service Worker
* เปิดแบบ Standalone และมีไอคอนบนหน้าจอโฮม
* Cache ทรัพยากรที่จำเป็นและ Offline Fallback
* คิวข้อมูลรอซิงก์เมื่อกลับมาออนไลน์
* Web Push Notification เมื่อ Browser และระบบปฏิบัติการรองรับ
* Camera, File Upload, Microphone และ Geolocation ตามสิทธิ์ของ Browser
* Deep Link ผ่าน URL และ Web Share API เมื่ออุปกรณ์รองรับ
* Progressive Enhancement และทางเลือกสำรองเมื่อความสามารถของ Browser ไม่พร้อมใช้งาน

ข้อจำกัดของ Phase 1 ต้องสื่อสารกับผู้ใช้อย่างชัดเจน โดยเฉพาะ Web Push, Background Sync, การแจ้งเตือนเมื่อ Browser ถูกปิด, การติดตามตำแหน่งต่อเนื่อง และการทำงานเบื้องหลัง ซึ่งอาจแตกต่างกันตามอุปกรณ์และระบบปฏิบัติการ

Phase 1 ไม่รวมการเผยแพร่ Native Application แยกบน App Store หรือ Google Play

## 12.3 Phase 2 — Cross-platform Mobile App

Phase 2 ต้องพัฒนา **Cross-platform Mobile Application สำหรับ Android และ iOS** โดยใช้ Backend, API, บัญชีผู้ใช้ และ Data Model เดียวกับ Phase 1

แพลตฟอร์มที่รองรับ ได้แก่

* Android Application ผ่าน Google Play หรือช่องทางทดสอบที่ได้รับอนุมัติ
* iOS Application ผ่าน Apple App Store หรือ TestFlight
* Smartphone เป็นอุปกรณ์หลัก
* Tablet ตามความต้องการของกลุ่มผู้ใช้และองค์กร
* Web Portal และ PWA เดิมยังคงใช้สำหรับผู้ดูแล แพทย์ ผู้ดูแลระบบ และผู้ใช้ที่ไม่ติดตั้งแอป

ความสามารถที่เพิ่มขึ้นใน Phase 2 ได้แก่

* Native Push Notification และ Local Notification
* Background Task และการจัดตารางงานในอุปกรณ์
* Biometric Authentication เช่น Fingerprint, Face ID หรือความสามารถเทียบเท่า
* Secure Storage สำหรับ Token และข้อมูลสำคัญ
* Offline Database แบบเข้ารหัสและการซิงก์ที่ทนทานขึ้น
* Widget, Shortcut และ Quick Action
* Universal Link, App Link และ Deep Link ระดับระบบ
* การทำงานร่วมกับ Camera, Microphone, Share Sheet และ File Picker ที่สม่ำเสมอขึ้น
* การเชื่อม Apple Health/HealthKit และ Android Health Connect เมื่อผู้ใช้ให้ความยินยอม
* การเชื่อมอุปกรณ์สุขภาพผ่าน Bluetooth เมื่ออยู่ในขอบเขตผลิตภัณฑ์
* Geofencing หรือการแชร์ตำแหน่งต่อเนื่องตามสิทธิ์และข้อกำหนดด้านความเป็นส่วนตัว
* Crash Reporting, Mobile Analytics และการจัดการเวอร์ชันแอป

## 12.4 ความต่อเนื่องระหว่างสองเฟส

เพื่อป้องกันการสร้างระบบซ้ำและลดความเสี่ยงในการย้ายข้อมูล ต้องยึดหลักดังนี้

* ใช้ Backend และ API Contract กลางตั้งแต่ Phase 1
* แยก Business Logic สำคัญออกจาก UI และ Browser-specific Code
* ใช้ Data Model, Validation Rule, Consent Model และ Permission Model ชุดเดียวกัน
* ผู้ใช้ Phase 1 ต้องเข้าสู่ Mobile App ด้วยบัญชีเดิมและเห็นข้อมูลเดิมได้
* การเปลี่ยนผ่านต้องไม่ทำให้ข้อมูลนัดหมาย ยา สุขภาพ เอกสาร หรือ Audit Log สูญหาย
* ใช้ Design System และภาษาการออกแบบเดียวกันระหว่าง PWA และ Mobile App
* Mobile App ต้องรองรับ API Versioning และนโยบายบังคับอัปเดตเฉพาะเมื่อจำเป็น
* PWA ต้องยังทำงานได้หลังเปิดตัว Phase 2 เพื่อรองรับผู้ใช้เดิมและบทบาทที่เหมาะกับหน้าจอเว็บ

## 12.5 หลักการเลือกเทคโนโลยี Cross-platform

การเลือก Framework สำหรับ Phase 2 ต้องประเมินจาก

* ความสามารถด้าน Notification และ Background Task
* การรองรับ HealthKit, Health Connect และ Bluetooth
* ความปลอดภัยของ Local Storage
* ความพร้อมของทีมพัฒนาและต้นทุนบำรุงรักษา
* ความสามารถในการใช้ Design System และ Business Rule ร่วมกัน
* ประสิทธิภาพบนอุปกรณ์ระดับกลางและระดับเริ่มต้น
* การรองรับ Accessibility และภาษาไทย
* อายุการสนับสนุนของ Framework และ Plugin สำคัญ

PRD ไม่บังคับ Framework เฉพาะ แต่ระบบต้องหลีกเลี่ยงการผูก Business Logic สำคัญไว้กับผู้ให้บริการหรือ Framework รายเดียวมากเกินไป

# 13. การทำงานแบบ Offline และการซิงก์ข้อมูล

## 13.1 Phase 1 — Offline ขั้นพื้นฐานสำหรับ PWA

Phase 1 ควรรองรับการใช้งาน Offline ขั้นพื้นฐาน โดยคำนึงถึงข้อจำกัดของ Browser และความปลอดภัยของข้อมูลสุขภาพ

ผู้ใช้ควรสามารถ

* เปิดหน้าที่เคยโหลดและได้รับอนุญาตให้ Cache
* ดูนัดหมายและรายการยาที่ซิงก์ไว้ล่าสุด
* ดูข้อมูลฉุกเฉินที่ผู้ใช้อนุญาตให้จัดเก็บบนอุปกรณ์
* ยืนยันการกินยาและเก็บรายการไว้ในคิวรอซิงก์
* บันทึกน้ำหนัก ความดัน น้ำตาล และชีพจรไว้ในคิวรอซิงก์
* ถ่ายรูปเอกสารหรือบันทึกเสียงไว้ชั่วคราวเมื่อ Browser รองรับ
* เปิดหมายเลขโทรศัพท์เพื่อโทรหาผู้ติดต่อฉุกเฉิน

## 13.2 Phase 2 — Offline-first สำหรับ Mobile App

Phase 2 ต้องเพิ่มความทนทานของการทำงาน Offline โดย

* ใช้ฐานข้อมูลภายในอุปกรณ์ที่เข้ารหัส
* รองรับการอ่านและแก้ไขข้อมูลสำคัญโดยไม่ต้องออนไลน์
* จัดคิว Upload รูปภาพ เสียง และเอกสาร
* ทำ Background Sync ตามข้อจำกัดของระบบปฏิบัติการ
* แสดงสถานะ Pending, Syncing, Synced และ Failed
* Retry แบบควบคุมได้และไม่สร้างข้อมูลซ้ำ
* รองรับการออกจากระบบพร้อมลบข้อมูล Local ตามนโยบาย
* รองรับ Remote Session Revocation และการป้องกันข้อมูลหลังอุปกรณ์สูญหายตามขอบเขตที่ระบบทำได้

## 13.3 กติกาการซิงก์ร่วมกัน

เมื่อกลับมาออนไลน์ ระบบทั้งสองเฟสต้อง

* ซิงก์ข้อมูลอัตโนมัติหรือให้ผู้ใช้กดซิงก์ได้
* แสดงรายการที่ยังไม่ซิงก์อย่างชัดเจน
* ใช้ Idempotency Key หรือกลไกเทียบเท่าเพื่อป้องกันข้อมูลซ้ำ
* ตรวจสอบ Version หรือ Updated Timestamp ก่อนเขียนทับข้อมูล
* จัดการ Conflict ตามประเภทข้อมูลและแสดงให้ผู้ใช้เลือกเมื่อจำเป็น
* เก็บ Audit Log ของการสร้าง แก้ไข ซิงก์ และแก้ Conflict
* แจ้งผู้ใช้เมื่อการซิงก์ล้มเหลวหรือข้อมูลอาจไม่เป็นปัจจุบัน

ข้อมูลเกี่ยวกับยา นัดหมาย และสิทธิ์การแชร์ต้องใช้กติกา Conflict ที่เข้มงวดกว่าข้อมูลทั่วไป และห้ามแก้ไขแบบเงียบเมื่อมีความขัดแย้ง

# 16. ข้อกำหนดที่ไม่ใช่ฟังก์ชัน

## 16.1 ประสิทธิภาพ

* หน้าหลักควรแสดงข้อมูลสำคัญภายใน 3 วินาทีภายใต้เครือข่ายปกติ
* การยืนยันกินยาและการทำเช็กลิสต์ต้องตอบสนองต่อผู้ใช้ทันที แม้รายการจะรอซิงก์
* การอัปโหลดรูปและเสียงต้องแสดง Progress และสามารถ Retry ได้
* Mobile App ใน Phase 2 ต้องเปิดหน้าหลักจากสถานะพร้อมใช้งานได้อย่างรวดเร็วบนอุปกรณ์ระดับกลาง
* API สำคัญต้องมี Timeout, Retry Policy และ Error Message ที่ผู้ใช้เข้าใจได้

## 16.2 ข้อกำหนดเฉพาะ Phase 1 — PWA

* ใช้ HTTPS ทุกสภาพแวดล้อมที่ให้บริการจริง
* มี Web App Manifest และ Service Worker
* รองรับการติดตั้งบนหน้าจอโฮมเมื่อ Browser รองรับ
* มี Offline Fallback และ Cache Strategy ที่เหมาะสมกับข้อมูลสุขภาพ
* ไม่เก็บข้อมูลสุขภาพสำคัญใน Client Cache โดยไม่มีการป้องกันที่เหมาะสม
* แสดงสถานะ Online, Offline และสถานะการซิงก์ข้อมูล
* ทดสอบ Chrome, Safari, Edge และ Firefox ตาม Browser Support Matrix
* รองรับ Touch Target, Safe Area และ Responsive Layout
* มีทางเลือกสำรองเมื่อ Web Push, Background Sync, Camera, Microphone หรือ Geolocation ใช้งานไม่ได้

## 16.3 ข้อกำหนดเฉพาะ Phase 2 — Cross-platform Mobile App

* รองรับ Android และ iOS เวอร์ชันขั้นต่ำตาม Mobile OS Support Policy ของโครงการ
* ใช้ Secure Storage สำหรับ Credential และ Token
* เข้ารหัส Local Database และข้อมูลอ่อนไหวที่จัดเก็บในอุปกรณ์
* รองรับ Native Push, Local Notification และการตรวจสอบสิทธิ์แจ้งเตือน
* รองรับ Background Task โดยไม่ละเมิดข้อกำหนดของระบบปฏิบัติการ
* รองรับ Deep Link, App Link หรือ Universal Link
* มี Crash Reporting, Performance Monitoring และ Mobile Release Tracking
* รองรับ Forced Update เฉพาะกรณี Security หรือ API Compatibility ที่จำเป็น
* ผ่านการทดสอบบนอุปกรณ์จริงหลายระดับ ไม่พึ่ง Emulator เพียงอย่างเดียว
* มีขั้นตอนลบข้อมูล Local เมื่อ Logout, Revoke Session หรือบัญชีถูกลบ

## 16.4 ความพร้อมใช้งาน

* เป้าหมายความพร้อมใช้งานของบริการกลางอย่างน้อย 99.5% ใน Phase 1
* ทบทวนเป้าหมายเป็น 99.9% ก่อนขยายเชิงองค์กรใน Phase 2 หากงบประมาณและ SLA รองรับ
* มีระบบสำรองข้อมูลและขั้นตอนกู้คืน
* เหตุการณ์สำคัญต้องมี In-app Reminder และช่องทางแจ้งเตือนตามความสามารถของแต่ละเฟส
* ระบบต้องแสดงสถานะบริการหรือข้อความเมื่อฟังก์ชันภายนอก เช่น OCR, Push หรือแผนที่ขัดข้อง

## 16.5 ความสามารถในการขยาย

* รองรับผู้ใช้จำนวนมาก หลายโรงพยาบาล และหลายองค์กร
* แยกข้อมูลแต่ละองค์กรและกำหนดสิทธิ์ตาม Tenant
* เพิ่มประเภทข้อมูลสุขภาพใหม่ได้
* เชื่อมต่อระบบภายนอกผ่าน Versioned API
* Backend ต้องรองรับทั้ง PWA และ Mobile App โดยไม่สร้าง API ซ้ำโดยไม่จำเป็น

## 16.6 Accessibility

* รองรับ Screen Reader
* รองรับ Dynamic Font Size
* มี Contrast และขนาดตัวอักษรเหมาะสม
* ปุ่มมีพื้นที่กดเพียงพอ
* ไม่ใช้สีเป็นวิธีสื่อสารเพียงอย่างเดียว
* ไม่บังคับใช้ท่าทางซับซ้อน
* Mobile App ต้องรองรับ Accessibility API ของ Android และ iOS

## 16.7 ความน่าเชื่อถือของข้อมูล

* เก็บเวลาที่สร้างและแก้ไข
* เก็บแหล่งที่มาของข้อมูล
* แยกข้อมูลที่ผู้ใช้กรอก ข้อมูลจากอุปกรณ์ และข้อมูลที่ AI สร้าง
* เก็บประวัติการแก้ไขข้อมูลสำคัญ
* การย้ายจาก PWA ไป Mobile App ต้องไม่เปลี่ยนความหมายหรือหน่วยของข้อมูล
* ข้อมูลจาก Health Platform หรือ Bluetooth ต้องระบุแหล่งอุปกรณ์และเวลาที่วัด

# 18. Phase 1 — ขอบเขต Mobile-first PWA

Phase 1 ต้องพิสูจน์ว่าผลิตภัณฑ์สามารถลดปัญหาพลาดนัด พลาดยา และช่วยให้ผู้ดูแลมีส่วนร่วมได้จริง โดยไม่ต้องรอการพัฒนา Native Mobile App

## 18.0 รูปแบบผลิตภัณฑ์ Phase 1

Phase 1 จะพัฒนาเป็น **Web App แบบ Mobile-first PWA** โดยให้สมาร์ตโฟนเป็นอุปกรณ์หลัก ผู้ใช้สามารถเปิดผ่าน Browser หรือเพิ่มแอปไว้บนหน้าจอโฮมได้ โดยไม่ต้องดาวน์โหลดจาก App Store หรือ Google Play

หลักการสำคัญ ได้แก่

* Mobile-first: งานหลักต้องทำได้สะดวกบนหน้าจอมือถือ
* Responsive: รองรับ Tablet และ Desktop Browser
* Installable: เพิ่มไอคอน PWA บนหน้าจอโฮมได้
* Offline-capable: ใช้งานข้อมูลสำคัญบางส่วนได้เมื่อไม่มีอินเทอร์เน็ต
* Progressive Enhancement: ใช้ความสามารถของอุปกรณ์เท่าที่ Browser รองรับและมีทางเลือกสำรอง
* Single Backend: ใช้ Backend/API เดียวที่พร้อมรองรับ Mobile App ใน Phase 2
* Data Continuity: บัญชีและข้อมูลทั้งหมดต้องนำไปใช้ต่อใน Phase 2 ได้

* ผู้ใช้ต้องตรวจสอบข้อมูลจาก OCR, Speech-to-Text และ AI ก่อนบันทึก

ข้อจำกัดเหล่านี้ต้องถูกวัดระหว่าง Pilot เพื่อใช้ตัดสินใจและจัดลำดับงานใน Phase 2

# 19. Phase 2 — Cross-platform Mobile App

Phase 2 มีเป้าหมายยกระดับ Mo-nut จากผลิตภัณฑ์ที่ผ่านการพิสูจน์บน PWA ไปสู่แอปมือถือที่เหมาะกับการใช้งานประจำวัน มี Notification ที่เชื่อถือได้ขึ้น ทำงาน Offline ได้ดีขึ้น และเชื่อมความสามารถของอุปกรณ์โดยตรง

## 19.1 ขอบเขตแพลตฟอร์ม

* Cross-platform Mobile App สำหรับ Android และ iOS
* เผยแพร่ผ่าน Google Play และ Apple App Store หรือช่องทางทดสอบก่อนเปิดตัว
* ใช้บัญชี Backend API และข้อมูลเดิมจาก Phase 1
* คง PWA และ Web Portal สำหรับผู้ใช้ที่ไม่ติดตั้งแอป ผู้ดูแล แพทย์ และผู้ดูแลระบบ

## 19.2 ฟังก์ชันหลักที่ต้องเพิ่มหรือยกระดับ

1. Native Push Notification และ Local Notification สำหรับนัดหมาย ยา และเช็กลิสต์
2. Background Task สำหรับจัดตาราง แจ้งเตือน และซิงก์ข้อมูลตามข้อจำกัดของ OS
3. Biometric Authentication และ Secure Storage
4. Offline Database แบบเข้ารหัส พร้อม Sync Queue และ Conflict Handling ที่แข็งแรงขึ้น
5. Widget, Shortcut หรือ Quick Action สำหรับยารอบถัดไป Quick Capture และ SOS
6. Deep Link, App Link และ Universal Link
7. Camera, Microphone, File Upload และ Share Sheet แบบ Native
8. แชร์ตำแหน่งหรือ Geofencing ตามความยินยอมและข้อกำหนดของระบบปฏิบัติการ
9. เชื่อม Apple Health/HealthKit และ Android Health Connect
10. เชื่อมอุปกรณ์สุขภาพผ่าน Bluetooth สำหรับอุปกรณ์ที่ผ่านการคัดเลือก
11. ติดตามยาคงเหลือและเตือนเติมยา
12. AI สรุปคำแนะนำและสร้างเช็กลิสต์จากเสียงหรือเอกสาร โดยต้องให้ผู้ใช้ยืนยัน
13. LINE หรือ SMS เป็นช่องทางสำรองสำหรับเหตุการณ์ที่กำหนด
14. แผนที่และสภาพจราจรละเอียดขึ้น รวมถึงการแจ้งเตือนเวลาออกเดินทาง
15. ปฏิทินครอบครัวและการจัดการผู้ป่วยหลายคนที่ดีขึ้น
16. รองรับหลายภาษาเมื่อผ่านการทดสอบเนื้อหาและ Accessibility
17. รายงานเชิงลึกและ Analytics สำหรับผู้ใช้/ผู้ดูแล
18. Crash Reporting, Performance Monitoring และ Mobile Release Management

## 19.3 การย้ายผู้ใช้จาก Phase 1

* ผู้ใช้เข้าสู่แอปด้วยบัญชีเดิม
* ข้อมูลนัดหมาย ยา สุขภาพ เอกสาร เช็กลิสต์ และสิทธิ์การแชร์ต้องปรากฏครบถ้วน
* ไม่บังคับให้ผู้ใช้สร้างโปรไฟล์ใหม่
* ต้องมีหน้าชี้แจงสิทธิ์ใหม่ เช่น Notification, Location, Bluetooth และ Health Data
* การขอสิทธิ์ต้องทำตามบริบท ไม่ขอทุกสิทธิ์พร้อมกันโดยไม่มีเหตุผล
* ผู้ใช้ยังสามารถใช้ PWA ต่อได้ และข้อมูลต้องซิงก์ข้ามช่องทาง

## 19.4 บทบาทแพทย์และองค์กรใน Phase 2

Mobile App เน้นผู้ป่วยและผู้ดูแลเป็นหลัก ส่วนแพทย์และองค์กรควรใช้ Web Portal ที่เหมาะกับหน้าจอขนาดใหญ่ โดยสามารถเพิ่ม

* Doctor Portal เต็มรูปแบบ
* ระบบจัดการผู้ป่วยหลายคน
* การดูรายงานและแนวโน้มที่ผู้ป่วยอนุญาต
* การสร้างเช็กลิสต์หรือเป้าหมายให้ผู้ป่วย
* Analytics สำหรับโครงการนำร่อง

ฟังก์ชันองค์กรขั้นสูงอาจจัดเป็นโครงการย่อยตามสัญญาและไม่ควรทำให้การเปิดตัว Mobile App ล่าช้า

# 24. จุดแตกต่างของผลิตภัณฑ์

หมอนัดไม่ได้เป็นเพียงแอปเตือนวันนัดหรือเตือนยา แต่รวมวงจรการดูแลไว้ในระบบเดียว

จุดแตกต่างประกอบด้วย

1. รองรับผู้ป่วย ผู้ดูแล และแพทย์
2. เปลี่ยนภาพใบนัดเป็นรายการนัด
3. แสดงรูปยาขณะเตือน
4. บันทึกเสียงแพทย์และเภสัชกร
5. เปลี่ยนเสียงเป็นบันทึกและเช็กลิสต์
6. บันทึกคำถามก่อนพบแพทย์
7. ติดตามคำแนะนำด้านการออกกำลังกายและสุขภาพ
8. เชื่อมข้อมูลกับนัดหมายแต่ละครั้ง
9. แสดงสภาพจราจรและช่วยวางแผนการเดินทาง
10. มีระบบ SOS และแชร์ตำแหน่ง
11. ออกแบบสำหรับผู้สูงอายุ
12. ให้ผู้ป่วยควบคุมสิทธิ์ข้อมูล

# 25. ตัวชี้วัดความสำเร็จ

## 25.1 North Star Metric

**จำนวนผู้ป่วยที่ทำกิจกรรมดูแลสุขภาพสำคัญครบตามแผนในแต่ละสัปดาห์**

กิจกรรมสำคัญหมายถึง

* ไปตามนัด
* ยืนยันการกินยา
* บันทึกข้อมูลสุขภาพ
* ทำเช็กลิสต์ตามคำแนะนำ
* แชร์ข้อมูลกับผู้ดูแล

## 25.2 Product Metrics

* จำนวนผู้สมัครใหม่
* จำนวนผู้ใช้งานรายวัน
* จำนวนผู้ใช้งานรายเดือน
* อัตราการสร้างนัดแรก
* อัตราการเพิ่มยา
* อัตราการเปิด Notification
* อัตราการเชิญผู้ดูแล
* จำนวนใบนัดที่สร้างผ่าน OCR
* จำนวนบันทึกเสียง
* จำนวนเช็กลิสต์ที่ทำสำเร็จ
* จำนวนรายงานที่แชร์ให้แพทย์
* Retention 30 วันและ 90 วัน

## 25.3 Outcome Metrics

* อัตราพลาดนัดลดลง
* อัตราพลาดยาลดลง
* ความสม่ำเสมอในการวัดความดันเพิ่มขึ้น
* ความสม่ำเสมอในการทำตามเช็กลิสต์เพิ่มขึ้น
* ผู้ดูแลรับรู้เหตุการณ์สำคัญเร็วขึ้น
* ผู้ป่วยเตรียมข้อมูลก่อนพบแพทย์ได้ดีขึ้น
* ความพึงพอใจของผู้สูงอายุเพิ่มขึ้น

## 25.4 เป้าหมายเบื้องต้นของ Phase 1

* ผู้ใช้สร้างนัดแรกได้ภายใน 5 นาที
* ผู้ใช้เพิ่มยาและตั้งเวลาได้ภายใน 5 นาที
* ผู้ใช้อย่างน้อย 70% เปิด Notification ตามช่องทางที่อุปกรณ์รองรับ
* ผู้ป่วยอย่างน้อย 40% เชิญผู้ดูแล
* การส่ง Notification ผ่านช่องทางที่รองรับสำเร็จตามเป้าหมายที่ทีมกำหนดและตรวจสอบได้
* ผู้ใช้สามารถบันทึกข้อมูลหลักได้ภายใน 3–5 ขั้นตอน
* คะแนนความพึงพอใจไม่น้อยกว่า 4 จาก 5
* มีข้อมูลเพียงพอเพื่อระบุว่าข้อจำกัดใดของ PWA ส่งผลต่อ Retention หรือความปลอดภัยของ Reminder

## 25.5 เป้าหมายเบื้องต้นของ Phase 2

* ผู้ใช้ Phase 1 สามารถเข้าสู่ Mobile App และเห็นข้อมูลเดิมโดยไม่มีข้อมูลสูญหาย
* Core Journey บน Android และ iOS มี Feature Parity ตาม Scope ที่กำหนด
* อัตราการเปิดและตอบสนองต่อการแจ้งเตือนยา/นัดดีขึ้นจาก Baseline ของ Phase 1
* Crash-free Session และความเสถียรผ่านเกณฑ์ Release Readiness ของโครงการ
* ผู้ใช้สามารถทำงานหลักแบบ Offline และซิงก์ภายหลังได้สำเร็จ
* คะแนนความพึงพอใจของ Mobile App ไม่น้อยกว่า 4 จาก 5 ในกลุ่มทดสอบ
* Retention 30 วันดีขึ้นเมื่อเทียบกับกลุ่มผู้ใช้ PWA ที่มีลักษณะใกล้เคียงกัน
* ผู้ใช้เข้าใจและควบคุมสิทธิ์ Notification, Location, Health Data และ Bluetooth ได้

ค่าตัวเลขเป้าหมายสุดท้ายของ Phase 2 ต้องกำหนดจาก Baseline ที่ได้จริงใน Phase 1 ไม่ควรคาดเดาก่อนมีข้อมูล Pilot

# 26. ความเสี่ยงและแนวทางรับมือ

## 26.1 OCR อ่านข้อมูลผิด

แนวทางรับมือ

* ให้ผู้ใช้ตรวจสอบ
* เก็บภาพต้นฉบับ
* ไม่เปลี่ยนข้อมูลสำคัญอัตโนมัติ
* แสดงระดับความมั่นใจของข้อมูล

## 26.2 Speech-to-Text เข้าใจศัพท์ทางการแพทย์ผิด

แนวทางรับมือ

* เก็บไฟล์เสียงต้นฉบับ
* ให้แก้ไขข้อความ
* เชื่อมข้อความกับตำแหน่งเวลาในเสียง
* เพิ่มพจนานุกรมศัพท์เฉพาะ

## 26.3 Notification ไม่ทำงานหรือทำงานไม่สม่ำเสมอ

แนวทางรับมือ

* Phase 1 ใช้ Web Push และ In-app Reminder เมื่อแพลตฟอร์มรองรับ
* ตรวจสอบสิทธิ์และความสามารถของ Browser พร้อมแจ้งข้อจำกัดแก่ผู้ใช้
* ใช้ช่องทางสำรอง เช่น Email, SMS หรือ LINE ตามระดับความสำคัญและงบประมาณ
* Phase 2 ใช้ Native Push และ Local Notification พร้อมตรวจสอบการตั้งค่าประหยัดพลังงาน
* บันทึก Delivery, Open และ Action Event เพื่อวัด Reliability จริง
* ไม่อ้างว่าการแจ้งเตือนมีความแน่นอน 100%

## 26.4 ผู้ป่วยกดยืนยันยาโดยไม่ได้กินจริง

แนวทางรับมือ

* ระบุว่าข้อมูลเป็นการรายงานโดยผู้ใช้
* ไม่ใช้เป็นหลักฐานยืนยันทางการแพทย์
* ให้ผู้ดูแลเพิ่มหมายเหตุได้

## 26.5 ข้อมูลสุขภาพรั่วไหล

แนวทางรับมือ

* Encryption
* Role-based Access Control
* Audit Log
* Security Testing
* Session Revocation
* Incident Response Plan
* การตรวจสอบสิทธิ์เป็นระยะ

## 26.6 ผู้สูงอายุใช้งานไม่เป็น

แนวทางรับมือ

* Elderly Mode
* Onboarding แบบง่าย
* ผู้ดูแลช่วยตั้งค่า
* คู่มือเสียงและวิดีโอ
* ทดสอบกับผู้สูงอายุจริง

## 26.7 ความเสี่ยงจาก SOS

แนวทางรับมือ

* ยืนยันก่อนส่ง
* แสดงสถานะการส่ง
* แสดงหมายเลขฉุกเฉินโดยตรง
* ระบุว่าแอปไม่ใช่ศูนย์ฉุกเฉิน
* มีทางเลือกโทรโดยไม่ต้องใช้อินเทอร์เน็ต

## 26.8 ความเสี่ยงด้านกฎหมายและการแพทย์

แนวทางรับมือ

* แยกผลิตภัณฑ์ออกจากการวินิจฉัย
* มีคำเตือนที่ชัดเจน
* ให้ผู้เชี่ยวชาญตรวจสอบเนื้อหา
* ประเมินข้อกำหนดทางกฎหมายก่อนเปิดตัว
* มีเอกสารความยินยอมและนโยบายความเป็นส่วนตัว


## 26.9 ความเสี่ยงจากการย้าย PWA ไป Mobile App

แนวทางรับมือ

* ใช้บัญชี Backend API และ Data Model เดียวกันตั้งแต่ Phase 1
* มี Migration Test และ Data Reconciliation
* ทดสอบบัญชีจริงที่มีข้อมูลหลายประเภทก่อนเปิดตัว
* รองรับ Rollback และเปิดให้ใช้ PWA ต่อระหว่างช่วงเปลี่ยนผ่าน
* ห้ามบังคับสร้างบัญชีหรือข้อมูลสุขภาพใหม่โดยไม่จำเป็น

## 26.10 ความเสี่ยงจาก App Store และ Mobile OS

แนวทางรับมือ

* ตรวจสอบนโยบาย App Store และ Google Play ก่อนพัฒนาฟีเจอร์สำคัญ
* ไม่พึ่ง Plugin ที่ไม่มีการดูแลหรือไม่รองรับ OS รุ่นใหม่
* มี Mobile OS Support Policy และแผนอัปเดต Dependency
* เตรียม Privacy Label, Data Safety และข้อความขอสิทธิ์ให้ครบถ้วน
* วางแผน Release Channel สำหรับ Internal Test, Beta และ Production

# 29. Roadmap เชิงผลิตภัณฑ์แบบ 2 เฟส

## กิจกรรมก่อนเริ่ม Phase 1

กิจกรรมต่อไปนี้เป็นงานเตรียมความพร้อม ไม่ถือเป็นเฟสผลิตภัณฑ์เพิ่มเติม

* สัมภาษณ์ผู้ป่วย ผู้ดูแล แพทย์ และเภสัชกร
* ทดสอบ Prototype กับผู้สูงอายุ
* ตรวจสอบกฎหมาย ความยินยอม และขอบเขตความรับผิดชอบ
* สรุป Business Model และกลุ่ม Pilot
* วาง Backend, API, Data Model, Security และ Design System ให้รองรับสองเฟส

## Phase 1 — Mobile-first PWA

### เป้าหมาย

เปิดผลิตภัณฑ์ใช้งานจริงได้เร็วและพิสูจน์ว่าฟังก์ชันนัดหมาย ยา สุขภาพ และผู้ดูแลแก้ปัญหาหลักได้

### ผลส่งมอบหลัก

* Mobile-first PWA สำหรับ Smartphone, Tablet และ Desktop
* ระบบผู้ป่วยและผู้ดูแล
* นัดหมาย ยา ข้อมูลสุขภาพ OCR Speech-to-Text เช็กลิสต์ และคำถามสำหรับแพทย์
* แผนที่ SOS รายงาน และ Doctor Lite Access
* Offline ขั้นพื้นฐาน Web Push และช่องทางสำรองตามขอบเขต
* Consent, Permission, Audit Log และ Security ขั้นพื้นฐาน
* Analytics เพื่อวัด Adoption, Retention และ Notification Reliability

### Gate ก่อนเริ่ม Phase 2

* Core Journey ผ่าน Acceptance Criteria ของ Phase 1
* ผู้ใช้กลุ่มเป้าหมายยืนยันคุณค่าของผลิตภัณฑ์
* มี Baseline ของ Retention, Notification และปัญหาการใช้งาน
* ไม่มี Critical Security หรือ Data Integrity Issue ที่ยังไม่แก้ไข
* Backend/API พร้อมรองรับ Mobile App และ Data Migration
* มีงบประมาณและแผนดูแล Android/iOS ระยะยาว

## Phase 2 — Cross-platform Mobile App

### เป้าหมาย

เพิ่มความเสถียรของการแจ้งเตือนและ Offline Experience เชื่อมความสามารถของอุปกรณ์ และยกระดับผลิตภัณฑ์ให้พร้อมขยายผู้ใช้

### ผลส่งมอบหลัก

* Android และ iOS Cross-platform Application
* Native Push, Local Notification และ Background Task
* Biometric, Secure Storage และ Offline Database แบบเข้ารหัส
* Widget/Shortcut, Deep Link และ Native Device Integration
* HealthKit, Health Connect และ Bluetooth ตามลำดับความสำคัญ
* การย้ายบัญชีและข้อมูลจาก Phase 1 แบบต่อเนื่อง
* PWA และ Web Portal ที่ทำงานร่วมกับ Mobile App
* Mobile Analytics, Crash Reporting และ Release Management

### Gate ปิด Phase 2

* Android/iOS ผ่าน Acceptance Criteria และ Security Review
* ผู้ใช้เดิมเข้าถึงข้อมูลครบถ้วนโดยไม่มี Data Loss
* Notification และ Offline Reliability ดีขึ้นจาก Baseline Phase 1
* แอปพร้อมเผยแพร่หรือผ่านช่องทาง Distribution ที่โครงการกำหนด
* ทีมมีขั้นตอน Support, Monitoring, Incident Response และ Version Update

รายการ Enterprise Integration หลังจากนี้ถือเป็น Backlog หรือโครงการแยก ไม่ใช่ Phase 3 ของแผนหลัก

# 30. เกณฑ์การยอมรับระดับผลิตภัณฑ์

## 30.1 เกณฑ์การยอมรับ Phase 1 — Mobile-first PWA

Phase 1 ถือว่าพร้อมสำหรับโครงการนำร่องเมื่อ

1. ผู้ป่วยสามารถสร้างบัญชีและโปรไฟล์ได้สำเร็จ
2. ผู้ป่วยสามารถสร้างนัดจากรูปใบนัดและยืนยันข้อมูล OCR ได้
3. ระบบส่ง In-app/Web Push หรือช่องทางแจ้งเตือนที่กำหนดได้
4. ผู้ป่วยสามารถเพิ่มยา รูปยา และตารางยาได้
5. ระบบแจ้งเตือนยาและบันทึกสถานะกินแล้ว ข้าม หรือเตือนภายหลังได้
6. ผู้ดูแลได้รับแจ้งเมื่อผู้ป่วยพลาดยาตามกติกาที่กำหนด
7. ผู้ป่วยบันทึกความดัน น้ำหนัก ชีพจร และน้ำตาลได้
8. ระบบแสดงประวัติและกราฟพื้นฐานได้
9. ผู้ป่วยบันทึกเสียงและแปลงเป็นข้อความได้ พร้อมแก้ไขก่อนบันทึก
10. ผู้ป่วยสร้างเช็กลิสต์และคำถามสำหรับแพทย์ได้
11. ผู้ใช้เปิดแผนที่นำทางได้
12. ผู้ใช้กด SOS โทรหาผู้ดูแล และส่งตำแหน่งปัจจุบันเมื่ออนุญาตได้
13. ผู้ป่วยควบคุมสิทธิ์แชร์ข้อมูลและเพิกถอนสิทธิ์ได้
14. ระบบสร้างรายงาน PDF หรือลิงก์แชร์แบบมีวันหมดอายุได้
15. งาน Offline ที่กำหนดสามารถเก็บคิวและซิงก์เมื่อกลับมาออนไลน์ได้
16. ระบบป้องกันข้อมูลซ้ำและแสดงสถานะการซิงก์ได้
17. ระบบผ่าน Security, Privacy และ Accessibility Test ขั้นพื้นฐาน
18. ผู้สูงอายุในกลุ่มทดสอบทำงานหลักได้ด้วยตนเองหรือด้วยความช่วยเหลือเล็กน้อย
19. ระบบเก็บ Analytics ที่จำเป็นต่อการตัดสินใจ Phase 2 โดยไม่เก็บข้อมูลเกินความจำเป็น
20. ไม่มี Critical Defect ที่ทำให้ข้อมูลยา นัดหมาย สิทธิ์การแชร์ หรือข้อมูลสุขภาพสูญหายหรือผิดบุคคล

## 30.2 เกณฑ์พร้อมเปลี่ยนผ่านสู่ Phase 2

* Core Journey ของ Phase 1 มีการใช้งานจริงและผ่านเกณฑ์คุณค่าที่ทีมผลิตภัณฑ์กำหนด
* มีข้อมูลเปรียบเทียบข้อจำกัดของ PWA ที่กระทบ Notification, Offline หรือ Retention
* API Contract, Authentication, Permission และ Data Model ผ่าน Architecture Review
* มี Test Data และ Migration Plan สำหรับผู้ใช้เดิม
* มี Mobile Security Requirement, OS Support Policy และ Store Compliance Checklist

## 30.3 เกณฑ์การยอมรับ Phase 2 — Cross-platform Mobile App

Phase 2 ถือว่าพร้อมเปิดตัวเมื่อ

1. Android และ iOS รองรับ Core Journey ตามขอบเขตที่อนุมัติ
2. ผู้ใช้ Phase 1 เข้าสู่ระบบด้วยบัญชีเดิมและเห็นข้อมูลเดิมครบถ้วน
3. ข้อมูลที่สร้างจาก PWA และ Mobile App ซิงก์ข้ามช่องทางได้โดยไม่สร้างข้อมูลซ้ำ
4. Native Push และ Local Notification ทำงานตาม Test Matrix รวมถึงกรณีแอปอยู่เบื้องหลัง
5. แอปแสดงคำแนะนำเมื่อผู้ใช้ปิด Notification หรือระบบจำกัดการทำงานเบื้องหลัง
6. Biometric และ Secure Storage ทำงานตามความสามารถของอุปกรณ์
7. งาน Offline ที่กำหนดทำได้และซิงก์ภายหลังโดยมี Conflict Handling
8. Camera, Microphone, File Upload, Deep Link และ Share Sheet ทำงานบนอุปกรณ์จริง
9. Health Data, Bluetooth หรือ Location ขอสิทธิ์ตามบริบทและสามารถเพิกถอนได้
10. แอปผ่าน Security Review, Privacy Review และ Mobile Accessibility Test
11. ไม่มี Critical Crash, Data Loss หรือ Permission Escalation Issue
12. Crash Reporting, Monitoring, Release Version และ Incident Response พร้อมใช้งาน
13. PWA และ Web Portal ยังคงใช้งานร่วมกับ Mobile App ได้
14. Build ผ่านช่องทางทดสอบหรือการเผยแพร่ที่โครงการกำหนดสำหรับทั้ง Android และ iOS
15. เอกสารผู้ใช้ นโยบายความเป็นส่วนตัว Store Disclosure และ Support Process พร้อมก่อนเปิดตัว

# 32. บทสรุปสำหรับลูกค้าและนักลงทุน

หมอนัดเป็นแพลตฟอร์มที่แก้ปัญหาการดูแลผู้ป่วยโรคเรื้อรังในชีวิตประจำวัน โดยเชื่อมโยงข้อมูลนัดหมาย ยา สุขภาพ คำแนะนำ และผู้ดูแลไว้ในระบบเดียว

กลยุทธ์การพัฒนาแบ่งเป็น 2 เฟสอย่างชัดเจน: **Phase 1 เปิดตัวเป็น Mobile-first PWA เพื่อทดสอบตลาดอย่างรวดเร็ว** และ **Phase 2 ต่อยอดเป็น Cross-platform Mobile App สำหรับ Android/iOS** เพื่อเพิ่มความเสถียรของ Notification, Offline, Biometric และการเชื่อมต่ออุปกรณ์ โดยใช้บัญชี Backend API และข้อมูลชุดเดียวกัน

ผลิตภัณฑ์มีโอกาสขยายจากแอปสำหรับผู้ป่วยทั่วไปไปสู่แพลตฟอร์มสำหรับครอบครัว โรงพยาบาล คลินิก ร้านขายยา บริษัทประกัน และหน่วยงานสาธารณสุข

ความได้เปรียบหลักประกอบด้วย

* ปัญหาที่แก้ไขเกิดขึ้นซ้ำและต่อเนื่อง
* ผู้ใช้งานมีความจำเป็นต้องกลับมาใช้เป็นประจำ
* สามารถสร้าง Network Effect ระหว่างผู้ป่วย ผู้ดูแล และสถานพยาบาล
* รองรับรายได้ทั้ง B2C และ B2B
* สามารถเริ่มจาก Phase 1 ที่มีต้นทุนควบคุมได้ และต่อยอดสู่ Mobile App ใน Phase 2 โดยไม่สูญเสียข้อมูลเดิม
* มีโอกาสสร้างข้อมูลสุขภาพที่มีคุณค่าโดยอยู่ภายใต้ความยินยอมและการควบคุมของผู้ป่วย
* ตอบโจทย์สังคมผู้สูงอายุและการดูแลผู้ป่วยโรคเรื้อรังระยะยาว

**หมอนัดไม่ได้เป็นเพียงแอปแจ้งเตือน แต่เป็นผู้ช่วยเชื่อมการดูแลผู้ป่วยระหว่างบ้าน ครอบครัว และสถานพยาบาลอย่างต่อเนื่อง**
